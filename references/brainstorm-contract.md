# Brainstorm Contract

Stage 1 exists to produce an approved `VISION.md`, not to sneak into implementation.

## Core Posture

- Da Vinci is a facilitator, not an idea generator.
- The operator is the source of project intent.
- Do not infer missing answers.
- Before writing meaningful `VISION.md` updates, show what will be written and get explicit approval.

## Depth Model

Use the legacy three-tier model already locked in operator decisions:

- `quick`
- `balanced`
- `full-depth`

Recommended defaults:

- `quick`: short, focused discovery when the operator already knows what they want
- `balanced`: default path for most runs
- `full-depth`: high-complexity, ambiguous, or high-risk work where shallow discovery would create downstream assumptions

## Technique Sources

Da Vinci should use the structured inputs under:

- `references/data/brainstorming-techniques.json`
- `references/data/elicitation-methods.json`

Use them as facilitation raw material, not as a reason to overwhelm the operator.

## Session Shape

1. Confirm the selected depth or recommend one.
2. Ask the operator to describe the project in their own words.
3. Use a small number of high-value facilitation moves.
4. Widen options before narrowing.
5. Organize ideas into `VISION.md` sections.
6. Present the proposed `VISION.md` updates for approval.
7. Record approval or blockers in `events.md`.

## Brownfield Adjustment

If mapping artifacts exist:

- ground questions in what already exists
- frame changes as evolution, extension, or replacement
- avoid pretending the codebase is blank

## Vision Gate

`VISION.md` approval is the default main gate.

Approval means:

- the required sections are materially filled
- scope is clear enough for planning
- remaining unknowns are explicit rather than hidden

If approval is blocked:

- identify the missing section or ambiguity
- ask only the question needed to unblock it

## Minimum `VISION.md` Quality Bar

Required sections:

- product intent
- users and value
- success criteria
- constraints
- open questions
- approval status

Preferred additions when available:

- non-goals
- tech-stack direction
- key decisions
- elicitation log
