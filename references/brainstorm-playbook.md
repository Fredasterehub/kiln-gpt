# Brainstorm Playbook

Kiln brainstorm is not "throw ideas at a wall and hope one survives."

It is a guided extraction pass that converts vague ambition into execution-grade intent.

## Operator Setup

Start by asking the operator to choose:

- `depth`: `light`, `standard`, or `deep`
- `style`: `express` or `tour`

Interpret them like this:

- `light`: fast shaping, only the most critical questions
- `standard`: default, enough rigor to build safely
- `deep`: product, risk, GTM, edge cases, failure modes, and deployment realities
- `express`: concise, low ceremony, skip long explanations
- `tour`: explain why each cluster of questions matters and summarize often

## Required Output Areas

Before the brainstorm is complete, capture all of these:

1. Problem worth solving
2. Who the user is
3. Core workflows
4. Success criteria
5. Non-goals
6. Constraints
7. Existing assets or brownfield realities
8. Technical preferences and prohibitions
9. Validation expectations
10. Deployment expectations
11. Open questions that genuinely block planning

## Technique Strategy

Do not dump a list of techniques on the operator. Use them selectively.

Pick 2 to 5 from the technique bank based on what is missing:

- if the user is vague, use framing and differentiation techniques
- if the user is ambitious, use scope compression and sequencing techniques
- if the user is product-heavy, use workflow and outcome techniques
- if the user is technical, use architecture and constraints techniques
- if the user is hand-wavy about launch, use deployment and operational techniques

## Completion Standard

The brainstorm is done only when:

- `vision.md` is concrete enough to plan from
- major contradictions are surfaced
- success and deployment are both testable
- unresolved questions are few and explicit
